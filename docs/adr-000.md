# ADR 000: Frontend TypeScript and Lit Migration Survey

Status: proposed
Date: 2026-05-23
Repository snapshot: `main` at `a5cf031d115741813a5536afea40bbb4941fe425`

## Context

ghrm currently uses Rust, Askama templates, htmx navigation, and feature-scoped JavaScript modules. The frontend is not a SPA. Rust renders the page structure, htmx swaps `article.markdown-body`, and JavaScript reattaches behavior after swaps.

The question under review is whether the UI should move toward TypeScript and a client-side framework, especially Lit, to improve long-term maintainability, testing, verifiability, and portfolio quality without diminishing the speed and clarity of the Rust backend.

The original prompt asked for an independent multi-agent survey rather than a default framework recommendation:

> i'm considering porting the ui over to typescript and/or a typescript framework. don't just say react, and don't just dismiss react because i said that.
>
> i want a truly intelligent answer on how much more my code will benefit 10 years from now by moving to a better client side framework and structure. my expectation is that i can then begin implementing ui side testing, verifiability, and all sorts of well-controlled, less chaos practices that can be afforded to it. clearly, i've done an okay job breaking pieces out on both the backend and frontend side. i want a frontend that's something i can be more proud of, shows true pedigree for the next job opportunities i can find, and give it the respect this lightning fast backend deserves.

The follow-up prompt requested a Lit-focused survey using several high-capability agents:

> you may use several gpt-5.5/high agents to conduct a survey on a lit rewrite. i want you to have them provide you a report on the complexity of this task and for them to determine, from the components/boundaries you have them look at, what the loc, pathway, etc cost of doing this is going to be.
>
> concepts to think about: your code as a crime scene, change coupling, complexity, madge is available, fitness functions, the strangler fig pattern, tokei is available, cargo-mutants will take 2 hours to finish--good to do this anyway, just not in their scope.
>
> tsc --allowJs --checkJs --noEmit assets/js/*.js provided a lot of stuff. etc.
>
> try and frame the agents so they can provide perspectives. by averaging, overlapping, and combining the results of a few independent agents, we can get a pretty good dial on what this will take.

## Current Architecture Findings

The code is not chaotic in the import-graph sense. The main risk is change coupling through implicit DOM contracts.

Important contracts:

- `article.markdown-body` is the htmx target and history element.
- `nav.js` is the effective lifecycle root because it handles htmx swaps and reruns setup functions.
- `search.js` replaces `<tbody>` with server-rendered HTML fragments, then reruns column/date/link setup.
- `explorer.js` owns column visibility, menu open/close, archive polling, date formatting, and external HTML links.
- `file.js` creates toolbar DOM client-side inside a server-rendered header.
- `gist.js` is partly outside htmx lifecycle because it fetches HTML, parses it, and replaces the article manually.
- feature adapters mutate markdown-rendered light DOM after render.

This means a framework rewrite that ignores the server/client contract would move bugs from plain JavaScript into component integration rather than removing them.

## Hotspots

`assets/js/gist.js`

- 623 lines.
- Highest state and behavior complexity.
- Owns editor state, save flow, live refresh, textarea/blob sync, row rename, URL replacement, wrap preference, and dirty-state protection.
- Highest-risk functions: `save`, `refreshArticle`, `setupGistEditor`, `saveRowRename`, `setupLiveGist`.

`assets/js/explorer.js`

- 357 lines.
- Owns explorer menu state, archive progress, column visibility, date formatting, and external link insertion.
- Highest-risk functions: `syncColumnControls`, `setupViewMenu`, `startArchiveJob`, `setupNavExternalLinks`.

`assets/js/search.js`

- 260 lines.
- Owns global topbar search state and server fragment insertion.
- Highest-risk behavior: replacing table rows while preserving column visibility, dates, external links, and dirty close behavior.

`assets/js/nav.js`

- 113 lines.
- Small but central.
- Owns htmx lifecycle and dispatches `ghrm:contentready`.

`assets/js/adapters/code.js`

- 190 lines.
- Contains highlighted-node line splitting. This is easy to regress with DOM rewrites.

## Decision Direction

Adopt TypeScript first and Lit islands second.

Do not start with a full frontend replacement. Do not move routing, view query semantics, table rendering, or markdown rendering into the client early. Rust and Askama should continue to own page structure, rendered content, search fragments, URL generation, and htmx full-page/fragment decisions.

Lit should be used as a typed island layer for interactive state that the browser naturally owns:

- transient popover state
- keyboard interactions
- localStorage preferences
- archive polling
- clipboard
- editor interactions
- date formatting
- post-swap enhancement

Lit should use light DOM for the initial migration. Shadow DOM would worsen coupling for this app because CSS, htmx, markdown renderers, highlight.js, Mermaid, KaTeX, map adapters, and TOC scanning all currently target or mutate light DOM.

## Candidate Lit Islands

Low-risk first candidates:

- `ghrm-archive-progress`: owns archive POST, polling, progress labels, and timers.
- `ghrm-explorer-menu`: owns menu open/close/position/Escape behavior while the server keeps links and active state.

Medium-risk candidates:

- `ghrm-file-view`: owns raw/preview toggle, wrap toggle, copy/download controls. Keep preview/raw panes as light-DOM slots because vendor renderers mutate them.
- `ghrm-toc-popover`: owns TOC button state and panel positioning, but still scans light DOM outside itself.
- `ghrm-gist-stash`: owns row rename and stash refresh.
- `ghrm-path-search`: owns open/mode/query/sequence state while preserving server-rendered search fragments.

Highest-risk candidate:

- `ghrm-gist-editor`: replaces most of `gist.js` editor state, save, wrap, indentation, blob sync, and dirty live-refresh handling. Split pure editor transforms out before converting.

Avoid early:

- Lit row/cell components inside `<table>`.
- JSON table rendering for explorer/search rows.
- TypeScript-side duplication of `src/explorer/view.rs` query and URL semantics.

## Strangler-Fig Migration Path

1. Add TypeScript checking around current JavaScript without changing runtime behavior.
2. Add ambient declarations for vendor globals and typed helpers for DOM selectors, htmx events, `ghrm:contentready`, `ghrm:themechange`, file view attrs, gist attrs, and `data-ghrm-features`.
3. Add Rust template contract tests for required selectors and attributes.
4. Convert pure modules first: `dom`, `live`, `prefs`, `vendor`.
5. Add fast DOM tests with happy-dom or jsdom for current behavior.
6. Convert `file.js` or archive progress as the first Lit island.
7. Convert explorer menus while preserving server-rendered table rows.
8. Split gist pure editor behavior from DOM behavior.
9. Convert gist stash rename, then gist editor.
10. Convert path/content search controller while preserving server-rendered fragments.
11. Consider JSON endpoints and client table rendering only after table behavior has contract and browser coverage.

## Build Tool Notes

Vite is the better fit if Lit islands are likely because it handles multi-entry browser builds and a future dev-server workflow cleanly.

esbuild is viable for a smaller TS-to-JS build path, but it does not type-check TypeScript, so `tsc --noEmit` must remain a separate check.

The Rust asset embedding path expects maintained runtime assets under `assets/css` and `assets/img`, plus packaged browser output under `assets/js.tar.zst` and `assets/js.sha256.json`:

- `build.rs` hashes `css`, `img`, `js.tar.zst`, and `js.sha256.json`.
- `src/http/assets.rs` embeds `assets/css`, `assets/img`, and the zstd JS archive.
- `ui/scripts/check.js` packs Vite output into the zstd archive, writes the SHA-256 manifest, extracts the archive during `build:check`, and compares it to fresh Vite output.
- `assets/config.json` remains the vendor asset source of truth.

Any build output path must respect this embedding and dev-watch model or intentionally revise it.

## Consequences

Positive:

- TypeScript can make the implicit client contracts explicit.
- Lit islands can improve state ownership without replacing the fast Rust/htmx shell.
- The migration can be staged and verified one behavior at a time.
- Portfolio value comes from typed contracts, tests, and measured migration, not from adopting a framework label.

Negative:

- Tooling must be added where none currently exists.
- The first real work is contract and test infrastructure, not visible UI changes.
- Lit components must be careful with htmx teardown and post-swap lifecycle.
- Shadow DOM is mostly a poor fit for the first migration phase.
- A full app shell would duplicate server ownership unless the project deliberately changes architecture.

## Current Recommendation

Proceed with a TypeScript-first, test-first, light-DOM Lit island migration.

Do not rewrite the frontend as a full client app at this stage. Keep Rust and Askama responsible for routing, htmx response shape, view query semantics, table rows, search fragments, rendered markdown/source HTML, and vendor feature declarations. Move browser-owned transient interaction state into typed modules and then into Lit islands where that reduces coupling.
